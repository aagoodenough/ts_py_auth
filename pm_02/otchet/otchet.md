# РАЗДЕЛ 1. АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ

## 1.1. Описание проекта и постановка задачи

В рамках данного проекта была разработана система аутентификации пользователей с поддержкой двух методов входа:
- традиционная регистрация и вход по email и паролю;
- OAuth 2.0 авторизация через внешние провайдеры (Google, GitHub).

Основное назначение системы — обеспечение безопасного доступа к защищённым ресурсам веб-приложения. Система реализует функции регистрации новых пользователей, аутентификации существующих пользователей, генерации и валидации JWT-токенов для поддержания сессии, а также привязки внешних OAuth-аккаунтов к существующим пользователям.

## 1.2. Требования к программному обеспечению

### 1.2.1. Функциональные требования

Система аутентификации должна обеспечивать выполнение следующих функций:

**Модуль регистрации:**
- регистрация нового пользователя по email и паролю;
- валидация пароля (минимум 8 символов, минимум 1 заглавная буква, минимум 1 специальный символ);
- проверка уникальности email;
- генерация JWT-токена после успешной регистрации.

**Модуль входа:**
- аутентификация по email и паролю;
- проверка корректности пароля с использованием bcrypt;
- генерация JWT-токена для поддержания сессии.

**Модуль OAuth:**
- авторизация через Google;
- авторизация через GitHub;
- привязка OAuth-аккаунта к существующему пользователю по email;
- автоматическое создание нового пользователя при первом OAuth-входе.

**Модуль управления пользователями:**
- получение информации о текущем пользователе;
- обновление данных пользователя;
- выход из системы (удаление токена).

### 1.2.2. Нефункциональные требования

**Безопасность:**
- хеширование паролей с использованием bcrypt;
- использование JWT-токенов с подписью HS256;
- время жизни токена — 1 час;
- защита от SQL-инъекций через ORM;
- CORS-политика для разрешённых источников.

**Производительность:**
- асинхронная работа с базой данных;
- подключение к Supabase PostgreSQL через asyncpg;
- время отклика не более 2 секунд для типичных операций.

**Надёжность:**
- обработка ошибок с возвратом понятных сообщений;
- логирование запросов.

**Совместимость:**
- поддержка современных браузеров;
- адаптивный интерфейс для мобильных устройств.

# РАЗДЕЛ 4. РЕАЛИЗАЦИЯ

## 4.1. Общее описание

Форма авторизации является точкой входа в систему для всех пользователей. Реализована с использованием FastAPI (бэкенд) и Next.js (фронтенд). Система поддерживает два метода аутентификации: email/пароль и OAuth 2.0 (Google, GitHub).

## 4.2. Технические параметры

| Параметр | Значение |
|----------|----------|
| Стек Frontend | Next.js 14, React 18, TypeScript |
| Стек Backend | FastAPI, Python 3.11+, SQLAlchemy async |
| База данных | Supabase PostgreSQL |
| Аутентификация | Email/Password + Google/GitHub OAuth |
| Хранение сессии | localStorage (JWT токен) |
| Хеширование паролей | bcrypt (через fastapi-users) |
| Валидация | Pydantic схемы |
| Защита от SQL-инъекций | SQLAlchemy ORM |
| Защита от XSS | React auto-escaping |

## 4.3. Описание страниц авторизации

### 4.3.1. Страница входа (/)

Страница входа содержит следующие элементы:
- поле Email (type="email", required);
- поле Пароль (type="password", required);
- кнопка «Sign In» для входа;
- кнопки «Continue with Google» и «Continue with GitHub» для OAuth;
- ссылка «Don't have an account? Sign up» на страницу регистрации.

При успешном входе токен сохраняется в localStorage, пользователь перенаправляется на /dashboard. При ошибке отображается красное сообщение под формой.

### 4.3.2. Страница регистрации (/register)

Страница регистрации содержит:
- поле Email (type="email", required);
- поле Пароль (type="password", required);
- поле Подтверждение пароля (type="password", required);
- кнопка «Sign Up» для регистрации;
- ссылка «Already have an account? Sign in» на страницу входа.

Валидация пароля: минимум 8 символов, минимум 1 заглавная буква, минимум 1 специальный символ. При успешной регистрации — редирект на /dashboard.

### 4.3.3. Защищённый дашборд (/dashboard)

Дашборд доступен только авторизованным пользователям. При отсутствии токена происходит автоматический редирект на страницу входа.

Отображаемая информация:
- Email пользователя;
- ID пользователя;
- Статус верификации;
- Флаг OAuth-пользователя;
- Кнопка «Sign Out» для выхода.

При выходе токен удаляется из localStorage, пользователь перенаправляется на страницу входа.

## 4.4. Правила валидации формы

| Поле | Правило | Сообщение об ошибке |
|------|---------|-------------------|
| Email | Обязательное поле, валидный формат | Некорректный email |
| Пароль | Обязательное поле, минимум 8 символов | Пароль должен содержать не менее 8 символов |
| Пароль | Минимум 1 заглавная буква | Пароль должен содержать хотя бы одну заглавную букву |
| Пароль | Минимум 1 специальный символ | Пароль должен содержать хотя бы один специальный символ |
| Подтверждение пароля | Должно совпадать с паролем | Пароли не совпадают |

## 4.5. Схема OAuth 2.0

Процесс OAuth 2.0 для Google:

1. Пользователь нажимает «Continue with Google».
2. Фронтенд запрашивает /auth/oauth/google.
3. Бэкенд возвращает URL для авторизации Google.
4. Браузер редиректит на Google.
5. Пользователь выбирает аккаунт и даёт разрешение.
6. Google редиректит на /auth/google/callback с code.
7. Бэкенд обменивает code на access_token.
8. Бэкенд получает данные пользователя от Google.
9. Бэкенд создаёт/обновляет пользователя в БД.
10. Бэкенд возвращает JWT токен.
11. Фронтенд сохраняет токен и редиректит на /dashboard.

## 6.1. Описание интерфейса

### Страница входа
